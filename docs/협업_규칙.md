# EgAg — 협업 규칙

---

## Git 브랜치 전략

### 브랜치 구조

```
main              ← 배포용 (송아가 관리, 직접 push 금지)
  └── develop     ← 개발 통합 브랜치 (여기로 PR 보내기)
        ├── feat/canvas-drawing      ← A 규원
        ├── feat/auth-login          ← B 송아
        ├── feat/explore-feed        ← C 상준
        └── feat/common-navbar       ← D 진희
```

### 규칙
- **main** — 송아가 관리. develop에서 안정화되면 머지
- **develop** — 모든 기능 브랜치가 여기로 합쳐짐
- **feat/기능명** — 각자 작업할 때 develop에서 따서 사용

### 작업 흐름

```
1. develop에서 새 브랜치 생성
   git checkout develop
   git pull origin develop
   git checkout -b feat/기능명

2. 작업 후 커밋
   git add .
   git commit -m "feat: 로그인 API 구현"

3. push
   git push origin feat/기능명

4. GitHub에서 develop으로 PR 생성

5. 송아가 코드 확인 후 머지
```

### 브랜치 이름 예시

| 담당 | 브랜치 예시 |
|------|-----------|
| A 규원 | `feat/canvas-drawing`, `feat/ai-stroke-agent` |
| B 송아 | `feat/auth-login`, `feat/payment-portone` |
| C 상준 | `feat/explore-feed`, `feat/artwork-detail` |
| D 진희 | `feat/common-navbar`, `feat/admin-page` |

## 커밋 메시지 규칙

### 형식
```
타입: 간단한 설명
```

### 타입
| 타입 | 언제 쓰는지 | 예시 |
|------|-----------|------|
| `feat` | 새 기능 추가 | `feat: 회원가입 API 구현` |
| `fix` | 버그 수정 | `fix: 로그인 시 토큰 만료 오류 수정` |
| `style` | UI/CSS 변경 | `style: 로그인 페이지 반응형 적용` |
| `refactor` | 코드 리팩토링 | `refactor: AuthService 메서드 분리` |
| `docs` | 문서 수정 | `docs: API 명세 업데이트` |
| `chore` | 설정, 빌드 관련 | `chore: eslint 설정 추가` |

### 예시
```
feat: 캔버스 드로잉 기능 구현
fix: 좋아요 토글 시 카운트 안 맞는 버그 수정
style: 탐색 피드 카드 레이아웃 수정
docs: 개발환경 셋업 가이드 추가
```

## API 요청/응답 규칙

### 공통 형식

**성공 응답**
```json
{
  "id": "uuid",
  "email": "user@example.com",
  "nickname": "이그에그"
}
```

**에러 응답**
```json
{
  "error": {
    "code": "TOKEN_INSUFFICIENT",
    "message": "토큰이 부족합니다"
  }
}
```

### HTTP 상태 코드
| 코드 | 의미 |
|------|------|
| 200 | 성공 |
| 201 | 생성 성공 |
| 400 | 잘못된 요청 |
| 401 | 인증 실패 (로그인 필요) |
| 402 | 토큰 부족 |
| 403 | 권한 없음 |
| 404 | 찾을 수 없음 |
| 409 | 중복 (이메일, 닉네임 등) |
| 500 | 서버 오류 |

### 인증이 필요한 API
```
Authorization: Bearer <accessToken>
```

- 프론트 개발 서버(`localhost:5173`)에서 `/api/*` 요청 → 자동으로 백엔드(`localhost:8080`)로 프록시됨
- 별도 CORS 설정 걱정 없이 그냥 `/api/auth/login` 이런 식으로 호출하면 됨

## 작업 시 주의사항

1. **다른 사람 폴더 수정 금지** — 본인 담당 패키지에서만 작업
2. **common/ 수정 시 팀원에게 알리기** — 공유 코드라 영향 범위가 넓음
3. **PR 올리기 전 빌드 확인** — `./gradlew build` / `npm run build` 통과하는지 체크
4. **충돌 발생 시** — 본인 브랜치에서 `git merge develop` 해서 먼저 해결 후 push
